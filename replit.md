# BGP Dashboard

## Overview
The BGP Dashboard is a comprehensive property management solution for Bruce Gillingham Pollard, consolidating operations, property tracking, news curation, and contact management into a single, intelligent platform. It features a native PostgreSQL CRM with integrated AI capabilities to enhance efficiency and provide market insights for property management in key London areas, aiming to replace disparate systems.

## File Storage
All uploaded files are stored under the `ChatBGP/` root folder (not `uploads/`). Subfolders: `chat-files`, `chat-media`, `doc-templates`, `hots`, `landlord-packs`, `legal-dd`, `marketing-files`, `investment-marketing`, `profile-pics`, `runs`, `smart-docs`, `templates`, `advice-engine`, `archivist-temp`, `sp-temp`, `transcribe-tmp`. The folder is gitignored. Files are also persisted in the `file_storage` DB table for durability.

**Team Property Folders**: Dedicated folders directly under `ChatBGP/` for each internal team: `London Leasing`, `National Leasing`, `Investment`, `Tenant Rep`, `Development`, `Lease Advisory`, `Office Corporate`, `Admin`. All authenticated users can browse and download files from any folder (cross-team collaboration). Upload and delete are restricted to the team's own folder (or admins for all folders). Managed from Settings page via the "Team Property Folders" section. API: `GET /api/team-folders`, `GET /api/team-folders/:folder/files`, `POST /api/team-folders/:folder/upload`, `GET /api/team-folders/:folder/download/:filename`, `DELETE /api/team-folders/:folder/:filename`.

## System Architecture
The application is built with a modern web stack: React, Vite, TypeScript, Tailwind CSS, and Shadcn UI for the frontend; Express.js and Node.js for the backend; and PostgreSQL with Drizzle ORM for data persistence. A native PostgreSQL CRM forms the core of the data architecture.

KYC is a manual sign-off checklist — not a hard gate on invoicing. Invoices are manually reviewed in Xero before sending.
- **WIP Report**: CRM deals are the single source of truth. The /api/wip endpoint builds WIP data directly from crm_deals (excluding Dead/Leasing Comps/Investment Comps), deriving stage from deal status (pipeline/wip/invoiced), pulling fee as amtWip, joining xero_invoices for invoice amounts/numbers, and enriching with wip_entries (imported spreadsheets) for month/invoiceNo/fiscalYear where matched by deal name. Orphaned spreadsheet entries (unmatched) are included separately. Access-controlled for senior users.
- **Invoicing lifecycle**: Deal reaches "Exchanged" or "Completed" → system auto-creates a DRAFT Xero invoice (with PO number if set) → bookkeeper/accountant reviews and approves in Xero (DRAFT → AUTHORISED) → "Sync Xero" button on WIP Report pulls latest statuses → deal auto-promoted to "Invoiced". The "Invoiced" status is always the *result* of Xero approval, never manually set as a trigger. Individual invoice syncs on the Deals page also auto-promote. Stage mapping: Exchanged/Completed = WIP (awaiting invoice approval); Invoiced/Billed = invoiced (confirmed by Xero). **PO Number**: Stored on crm_deals.po_number, editable on deal detail form and HOTs form, included in Xero invoice Reference field. **KYC Gate (from 1 May 2025)**: Invoice creation (both manual and auto) is blocked until KYC is approved on the deal. KYC approval is a separate sign-off: once all Companies House checks pass, an "Approve KYC" button appears on the KYC panel. Records who approved and when (kycApprovedBy, kycApprovedAt). Old WIP statuses (DRAFT INV, DRAFT PO, AWAIT PO) are mapped to "Invoiced" on import.

## Performance & Reliability Improvements (March 2026)
- **Error Boundaries**: React error boundaries wrap all route groups (main app, addins) to catch rendering crashes with a recovery UI instead of blank screens.
- **Auto-refresh on focus**: Query client uses 5-minute staleTime with refetchOnWindowFocus, so data refreshes when users tab back to the app.
- **Smart retry**: GET requests retry up to 2 times on transient failures (skips 401/403/404). Mutations never retry.
- **Server-side pagination**: CRM list endpoints (companies, contacts, properties, deals) support optional `page` and `limit` query params. When provided, returns `{ data, total }` instead of a flat array.
- **SQL duplicate scanning**: Duplicate detection uses SQL GROUP BY/HAVING instead of loading all records into memory.
- **AI context caching**: buildSystemPrompt (10min TTL), getBusinessLearningsContext (5min TTL), and getAvailableTools (10min TTL) are cached in-memory. Cache is invalidated when learnings are created/updated/deleted.
- **Connection status banner**: Displays a non-intrusive top banner when the browser goes offline or the WebSocket connection drops, with a brief "Connection restored" confirmation on reconnect.
- **Cross-linking improvements**: WIP report deal refs link to deal detail pages. Dashboard diary events show CRM entity badges (property/deal) matched by subject text. Leasing schedule property headers link to property detail pages. Leads page shows matching CRM contacts with badges and offers a "Convert to Contact" action. Requirements (leasing & investment) show matched CRM properties based on requirement locations.
- **API rate limiting**: Login routes (20 req/15min) and general API routes (200 req/min) are rate-limited via `express-rate-limit`. Chat BGP endpoints (`/api/chat`, `/api/ai/`, `/api/chatbgp`) are excluded from rate limiting.
- **Bulk Apollo enrichment batching**: The `/api/apollo/bulk-enrich` endpoint processes contacts in configurable batches (1-50, default 25) with offset pagination (`hasMore`/`nextOffset`/`totalEligible`), preventing timeouts on large datasets. The frontend enrichment page loops through batches automatically with progress display.
- **Dashboard code splitting**: `dashboard.tsx` (originally ~3500 lines) is split into modular components under `client/src/components/dashboard/`: types, helpers, MyLeadsWidget, WipDashboardCard (with WipMiniSummaryTable/WipMiniFilterSection), AvailableUnitsWidget, InvestmentTrackerWidget, WidgetPickerDialog (with registries).
